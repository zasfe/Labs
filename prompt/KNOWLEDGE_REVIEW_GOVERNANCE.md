# 운영지식 검토·승인·정비 절차
## 목적
이 문서는 `SUBAGENT_KNOWLEDGE.md`를 대체하지 않고 보완한다.
서브에이전트가 생성한 사례·후보·패턴을 주기적으로 검토하여 다음 상태로 관리하기 위한 절차다.
```text
수집
→ 구조화
→ 검토
→ 승인 또는 보류
→ 기준 정의 반영
→ 재검증
→ 폐기 또는 유지
```

핵심 목적은 다음과 같다.

* 잘못된 지식의 기준 정의 편입 방지
* 중복 객체·관계·행동의 누적 방지
* 사례와 일반화의 분리 유지
* 근거·반례·예외의 지속적 보강
* 오래된 지식의 정기 정비
* 승인 책임과 변경 이력의 명확화

# 운영 단위

운영지식 저장소는 원본 입력 단위를 기준으로 사례(Case)를 생성한다.

원칙은 다음과 같다.

| 입력 | 생성 단위 |
|------|----------|
| 메일 1건 | Case 1개 |
| Mattermost 스레드 1개 | Case 1개 |
| Slack 스레드 1개 | Case 1개 |
| 회의록 1건 | Case 1개 |
| 작업보고서 1건 | Case 1개 |
| 장애보고서 1건 | Case 1개 |

여러 입력을 하나의 Case로 병합하지 않는다.

하나의 입력 안에 여러 사건이 존재하더라도 원본 단위를 유지한다.

관련성은 Case 병합이 아니라 `related_cases`를 통해 연결한다.

Pattern, Workflow, Rule은 여러 Case를 기반으로 생성하며, 하나의 Case만으로 일반화하지 않는다.

⸻

역할 분리

서브에이전트

다음 작업만 수행한다.

* 사례 구조화
* 기존 정의 검색
* 신규 후보 제안
* 근거·반례·예외 정리
* 검토 필요 항목 등록
* 승인된 정의 재사용

다음 작업은 수행하지 않는다.

* 후보 승인
* 기준 정의 직접 변경
* 기존 정의 삭제
* 파일 구조 변경
* 규칙 자동 실행 결정

검토자

메인 에이전트 또는 지정된 사람이 담당한다.

* 근거 검토
* 중복·충돌 검토
* 적용 범위 판단
* 후보 상태 결정
* 기준 정의 반영 승인
* 재검토 시점 지정

승인자

운영 영향이 있는 정의에 대한 최종 책임자다.

다음 항목은 승인자 확인이 필요하다.

* 새로운 객체 유형
* 관계 의미 또는 방향 변경
* 운영 행동 정의
* 워크플로우
* 규칙
* 기존 정의의 폐기
* 자동화 연결 가능성이 있는 변경

검토자와 승인자는 동일인일 수 있으나 승인 기록은 별도로 남긴다.

1. 주기별 운영 절차

1.1 작업별 검토

각 사례 생성 또는 수정 직후 수행한다.

검토 항목

* 원본 참조 존재 여부
* 주요 사실의 근거 연결 여부
* 객체와 관계의 기존 정의 재사용 여부
* 사실·추론·결정·행동·결과·검증 분리 여부
* 임시 식별자 표시 여부
* 후보 생성 필요성
* 검토 요청 누락 여부

처리 결과

다음 중 하나로 처리한다.

결과	처리
이상 없음	사례 유지
경미한 누락	사례 수정
근거 부족	NEEDS_EVIDENCE
판단 필요	06-review/pending/ 등록
기존 정의와 충돌	즉시 검토 요청
잘못된 구조화	수정 후 재검토

1.3 월간 승인 검토

매월 1회 수행한다.

대상

* 객체 후보
* 관계 후보
* 행동 후보
* 검증 가능한 패턴
* 워크플로우 후보
* 규칙 후보
* 폐기 또는 대체가 필요한 기존 정의

월간 검토 순서

후보 확인
→ 기존 정의 비교
→ 관련 사례 검토
→ 반례·예외 검토
→ 운영 영향 평가
→ 상태 결정
→ 승인 시 기준 정의 반영
→ 변경 이력 기록

상태 결정

상태	의미
ACCEPTED	기준 정의 반영 승인
NEEDS_EVIDENCE	추가 근거 필요
DEFERRED	시기 또는 범위 문제로 보류
REJECTED	후보 부적합
DEPRECATED	기존 정의 폐기 예정
ACTIVE	승인 후 현재 사용 중

승인되지 않은 후보를 ACTIVE로 변경하지 않는다.

1.4 분기별 정비

분기 1회 수행한다.

검토 대상

* 사용되지 않는 정의
* 중복 또는 유사 정의
* 오래된 패턴
* 예외 누적이 많은 규칙
* 임시 식별자가 남아 있는 객체
* 폐기 후보
* 색인과 실제 파일의 불일치
* 깨진 참조 경로
* 과도하게 커진 기준 파일

정비 항목

* 중복 통합
* 용어 표준화
* 별칭 추가
* 잘못된 관계 방향 교정
* 폐기 상태 지정
* 대체 정의 연결
* 오래된 후보 정리
* 패턴 재검증
* 메모리 승격 또는 제거
* 파일 분할 필요성 검토

파일 분할은 다음 조건 중 하나 이상일 때 검토한다.

* 정의 100개 초과
* 파일 1,500줄 초과
* 검색 충돌 반복
* 수정 충돌 반복
* 특정 도메인이 전체의 40% 이상
* 독립 검토가 필요한 영역 발생

2. 후보 검토 기준

2.1 객체 후보

다음 항목 확인이다.

* 독립적인 식별 필요성
* 반복 참조 여부
* 자체 속성 존재 여부
* 관계 또는 상태 보유 여부
* 행동의 대상 여부
* 기존 객체 유형으로 표현 가능한지 여부
* 단순 속성이나 값에 불과하지 않은지 여부

승인 금지 조건

* 단일 로그 메시지
* 일회성 메트릭
* 단순 제품명
* 상태 코드
* 기존 객체의 별칭
* 기존 객체의 속성으로 충분한 항목

⸻

2.2 관계 후보

다음 항목 확인이다.

* 주체와 대상 유형의 명확성
* 관계 방향의 명확성
* 기존 관계와 의미 중복 여부
* 시간 속성 필요 여부
* 근거 사례 존재 여부
* 문장 표현이 아닌 독립 관계로 관리할 가치

승인 금지 조건

* 근거 없는 의존성
* 단순한 동일 위치
* 일회성 동시 발생
* 원인으로 확인되지 않은 상관관계
* RELATED_TO로 충분한 임시 연결

⸻

2.3 행동 후보

다음 항목 확인이다.

* 수행 대상 명확성
* 실제 실행 가능한 단일 행동인지 여부
* 입력과 실행 주체 존재 여부
* 예상 결과와 검증 방법 존재 여부
* 기존 행동의 조합 또는 별칭인지 여부
* 운영 권한과 승인 절차 필요 여부

승인 금지 조건

* 모호한 행동명
* 결과를 행동으로 표현한 항목
* 여러 단계를 하나의 행동으로 합친 항목
* 실행 증적이 없는 가정된 행동
* 기존 행동의 표현 차이

⸻

2.4 패턴 후보

다음 항목 확인이다.

* 관련 사례 2건 이상
* 공통 조건 존재
* 공통 행동 또는 결과 존재
* 차이점 기록
* 실패 사례 확인
* 반례·예외 확인
* 적용 범위 명시

패턴은 관찰된 반복 구조이며 규칙이 아니다.

반례가 존재해도 패턴은 유지할 수 있으나 적용 범위를 좁혀야 한다.

⸻

2.5 워크플로우 후보

다음 항목이 모두 확인되어야 한다.

* 둘 이상의 행동
* 시작 조건
* 행동 순서
* 단계별 대상
* 전이 조건
* 종료 또는 검증 조건
* 실패 또는 예외 경로
* 복수 사례

행동 순서가 사례마다 크게 다르면 워크플로우가 아닌 패턴으로 유지한다.

⸻

2.6 규칙 후보

다음 항목이 모두 확인되어야 한다.

* 적용 객체
* 입력 조건
* 판단 결과 또는 기대 행동
* 측정 단위와 기준
* 근거 사례
* 반례와 예외
* 적용 제외 범위
* 운영 영향 평가
* 승인자

상관관계를 인과 규칙으로 승인하지 않는다.

규칙 승인 후에도 자동 실행은 별도 승인 대상으로 한다.

3.3 승인 후 반영

승인된 후보는 다음 순서로 반영한다.

1. 대상 기준 파일 수정
2. 안정적인 식별자 부여
3. 후보 파일에 승인 결과 기록
4. 검토 문서를 accepted/로 이동
5. 관련 사례의 후보 참조를 정식 정의 참조로 변경
6. INDEX.md 갱신
7. CHANGELOG.md 갱신
8. 영향받는 문서의 참조 확인

후보 파일을 즉시 삭제하지 않는다.

승인 기록과 이전 상태를 추적할 수 있도록 유지 또는 보관한다.


Case 문서는 승인 대상이 아니다.

Case는 원본 기록과 구조화 결과를 보존하는 역할이며 수정될 수 있다.

승인 대상은 Case에서 제안된 Object, Relationship, Action, Workflow, Rule이다.

⸻

3.4 거절 또는 보류 처리

거절

* 거절 이유 기록
* 재제안 조건 기록
* 06-review/rejected/로 이동
* 후보 상태를 REJECTED로 변경
* 관련 사례에서 거절 사실 참조

보류

* 부족한 근거 기록
* 재검토 조건 기록
* 재검토 예정일 지정
* 06-review/deferred/로 이동
* 후보 상태를 DEFERRED 또는 NEEDS_EVIDENCE로 변경

⸻

4. 기존 정의 변경 절차

기존 정의의 의미 변경은 신규 후보 승인과 동일한 수준으로 검토한다.

변경 유형

변경 유형	처리
오탈자 수정	경미 변경
설명 보강	경미 변경
별칭 추가	경미 변경
속성 추가	영향 검토
허용 관계 변경	승인 필요
관계 방향 변경	승인 필요
행동 입력 변경	승인 필요
규칙 조건 변경	승인 필요
식별자 변경	원칙적으로 금지
정의 삭제	폐기 절차 사용

폐기 형식

status: DEPRECATED
deprecated_at:
deprecated_by:
reason:
replaced_by:
affected_cases:

기존 정의는 삭제보다 폐기 상태 사용이 우선이다.

⸻

5. 장기 지식 승격 절차

Memory는 Case를 승격하는 것이 아니다.

여러 Case에서 반복적으로 검증된 운영지식만 Memory로 승격한다.

07-memory/에는 검증된 지식만 승격한다.

승격 조건

다음 조건을 모두 만족해야 한다.

* 복수 사례에서 확인
* 관련 근거 존재
* 반례와 예외 검토
* 일정 기간 유효성 확인
* 검토자 승인
* 적용 범위 명시
* 기존 지식과 충돌 없음

승격 대상

* 운영 원칙
* 반복 검증된 교훈
* 알려진 예외
* 장기적으로 유효한 주의사항

승격 제외

* 단일 장애 결론
* 임시 해결책
* 미확인 원인
* 특정 시점의 상태
* 개인 의견
* 승인되지 않은 규칙
* 재현되지 않은 패턴

⸻

6. 품질 지표

월간 또는 분기별로 다음 항목을 확인한다.

지표	목적
원본 연결률	사실 추적 가능성
증적 연결률	주장 검증 가능성
기존 정의 재사용률	중복 억제
임시 식별자 잔존 수	식별 품질
후보 승인률	후보 품질
NEEDS_EVIDENCE 장기 체류 수	근거 부족 관리
중복 후보 수	용어 및 검색 품질
반례 포함률	성급한 일반화 방지
깨진 참조 수	저장소 무결성
미분류 Inbox 수	처리 지연 관리
폐기 정의 참조 수	전환 상태 관리

지표 수치만으로 품질을 판단하지 않는다.

급격한 변화나 반복 오류의 탐지 용도로 사용한다.

⸻

7. 자동 점검 권장 항목

가능한 경우 다음 항목을 스크립트 또는 정기 작업으로 점검한다.

* 중복 식별자
* 존재하지 않는 파일 참조
* 잘못된 상대 경로
* 필수 메타데이터 누락
* 허용되지 않은 상태값
* 후보의 기준 파일 직접 삽입
* 승인 기록 없는 ACTIVE 상태
* 장기 미처리 검토 항목
* 장기 미분류 Inbox
* 폐기 정의를 신규 사례에서 사용
* 관련 사례가 없는 패턴·규칙
* 증적 없는 중요 사실

자동 점검은 오류 후보를 보고하는 역할이다.

자동 수정 또는 자동 승인은 수행하지 않는다.

⸻

8. 검토 우선순위

다음 순서로 우선 처리한다.

1. 운영 행동이나 자동화에 영향을 주는 규칙
2. 보안·권한·개인정보 관련 항목
3. 기존 정의와 충돌하는 후보
4. 여러 사례에서 반복 사용되는 신규 객체·관계·행동
5. 오래된 NEEDS_REVIEW
6. 근거가 보강된 워크플로우·규칙 후보
7. 용어와 별칭 정리
8. 낮은 영향도의 문서 표현 개선

⸻

9. 변경 완료 기준

검토 또는 승인 작업은 다음 조건을 만족해야 완료다.

* 후보 또는 변경 대상 식별자 확인
* 기존 정의와 비교 완료
* 관련 사례와 원본 확인
* 근거·반례·예외 확인
* 운영 영향 평가
* 검토 결과와 이유 기록
* 승인자 기록
* 기준 파일 반영 또는 상태 변경
* 후보와 검토 문서 상태 일치
* 관련 사례 참조 갱신
* INDEX.md 갱신
* CHANGELOG.md 갱신
* 깨진 참조 여부 확인
* 재검토 필요 시 일정 기록

하나라도 충족하지 못하면 완료 상태로 처리하지 않는다.

⸻

10. 운영 원칙 요약

서브에이전트는 구조화와 후보 제안
검토자는 근거와 충돌 검토
승인자는 기준 정의 반영 결정
사례는 사실 보존
패턴은 반복성 기록
규칙은 승인 후 사용
자동화는 별도 승인
삭제보다 폐기
확정보다 추적 가능성 우선
