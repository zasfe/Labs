# Hyper-Waterfall 문서 체계 가이드

> 이 문서는 로젝트의 문서 구조가 **왜 이렇게 설계되었는지**, 그리고 **어떻게 AI 페어프로그래밍의 품질을 보장하는지** 설명한다.

---

## 이 프로젝트는 무엇인가

프로젝트의 궁극적 목표는 **Hyper-Waterfall 방법론의 실증**이다. 코드는 결과물이고, **그 코드가 만들어진 과정의 기록**이 방법론의 증거이다.

---

## 왜 문서가 이렇게 많은가

`mydocs/` 폴더에는 729개 이상의 문서가 있다. 이것은 관료주의가 아니라, AI 페어프로그래밍에서 **품질을 보장하는 유일한 방법**이기 때문이다.

### AI의 근본적 한계

AI 페어프로그래머에는 두 가지 한계가 있다:

1. **세션이 끊기면 기억이 사라진다** — 어제 무엇을 했는지, 왜 그렇게 결정했는지 모른다
2. **자기 교정이 불가능하다** — 스스로의 판단이 맞다고 확신하며, 잘못된 방향으로 계속 진행한다

문서 체계는 이 두 가지 한계를 극복하는 장치이다.

---

## 7개 폴더, 각각의 역할

```
mydocs/
├── orders/           ← 오늘 무엇을 할 것인가
├── plans/            ← 어떻게 할 것인가
├── working/          ← 어떻게 했는가
├── report/           ← 결과가 무엇인가
├── feedback/         ← 무엇이 틀렸는가
├── tech/             ← 무엇을 발견했는가
└── troubleshootings/ ← 어디서 넘어졌는가
```

### orders/ — 오늘 할일

매일 작성하는 작업지시서이다. 마일스톤(M100)과 GitHub Issue 번호로 타스크를 관리한다.

**역할**: 새 세션에서 AI가 "지금 무엇을 해야 하는가"를 즉시 파악할 수 있다.

```markdown
# 오늘 할일 - 2026년 4월 9일

| Issue | 타스크 | 상태 |
|-------|--------|------|
| #86 | HWP 썸네일 자동 추출 | 진행중 |
```

### plans/ — 수행계획서 + 구현계획서

타스크를 시작하기 전에 **무엇을 할 것인지**(수행계획서)와 **어떻게 할 것인지**(구현계획서)를 작성한다. 작업지시자의 승인 후에만 구현을 시작한다.

**역할**: AI가 잘못된 방향으로 코드를 쌓기 전에, 계획 단계에서 방향성 오류를 잡는다.

| 문서 | 파일명 | 내용 |
|------|--------|------|
| 수행 계획서 | `task_m100_{번호}.md` | 목표, 범위, 예상 단계 |
| 구현 계획서 | `task_m100_{번호}_impl.md` | 구체적 코드 변경 계획, 파일별 수정 내용 |

### working/ — 단계별 완료보고서

구현을 3~6단계로 나누어 진행하며, 각 단계가 끝날 때마다 완료보고서를 작성한다. 작업지시자가 승인한 후에만 다음 단계로 넘어간다.

**역할**: 단계마다 품질 게이트를 거치므로, 잘못된 방향으로 멀리 가기 전에 교정할 수 있다.

```
task_m100_86_stage1.md  ← 1단계 완료: WASM API 구현
task_m100_86_stage2.md  ← 2단계 완료: CLI 명령 추가
task_m100_86_stage3.md  ← 3단계 완료: Chrome 확장 연동
```

### report/ — 최종 결과보고서

타스크가 완료되면 최종 보고서를 작성한다. 계획과 결과를 비교하고, 차이가 있으면 그 이유를 기록한다.

**역할**: 계획-실행-검증의 사이클을 닫는다. 계획과 결과의 차이가 다음 타스크의 계획 정확도를 높인다.

### feedback/ — 피드백 기록

작업지시자가 AI의 결과물을 검토하고 교정한 내용을 기록한다. 이것은 **AI가 절대 스스로 만들 수 없는 문서**이다.

**역할**: 인간의 도메인 지식이 AI에게 전달된 순간을 보존한다.

예시:
```
"pi=278을 pi=277만큼 계산하고 pi=279의 y가 시작되면 차이가 나지 않을 것 같아요"
```

이 한 줄의 피드백이 없었으면 AI는 잘못된 페이지네이션 계산을 계속 "개선"했을 것이다.

### tech/ — 기술 사항 정리

개발 과정에서 발견한 기술적 사실을 기록한다. HWP 포맷의 LINE_SEG 구조, TAC 테이블의 고정값 줄간격 처리 방식 등.

**역할**: AI의 학습 결과를 영구화한다. 문서가 없으면 다음 세션에서 같은 기술적 조사를 처음부터 반복한다.

### troubleshootings/ — 트러블슈팅

해결하기 어려웠던 문제와 그 해결 과정을 기록한다. 21개의 트러블슈팅 문서가 축적되어 있다.

**역할**: "이 함정에 다시 빠지지 마라"는 경고. 실패의 기록이 가장 값진 자산이다.

---

## 타스크 진행 절차

모든 타스크는 아래 절차를 **엄격하게** 따른다. 단계를 건너뛰지 않는다.

```
1. GitHub Issue 등록
2. 오늘할일(orders/) 기록
3. 타스크 브랜치 생성 (local/task{번호})
4. 수행계획서 작성 → [작업지시자 승인]
5. 구현계획서 작성 → [작업지시자 승인]
6. 단계별 구현
7. 단계별 완료보고서 → [작업지시자 승인]
8. (다음 단계 반복)
9. 최종 결과보고서 → [작업지시자 승인]
10. 오늘할일 상태 갱신
```

각 `[승인]` 지점이 품질 게이트이다. 코드 리뷰만으로는 잡을 수 없는 **방향성 오류**를 문서 리뷰에서 잡는다.

---

## 왜 이것이 3곳에서 이동하며 작업할 수 있게 하는가

이 프로젝트의 메인테이너는 사무실, 집, 이동 중 — 3곳에서 각각 다른 클로드 세션으로 작업한다. 매번 새 세션은 이전의 기억이 없다.

하지만 문서가 있으면:

| "지금 뭘 해야 하지?" | → `orders/20260409.md` |
|---|---|
| "어디까지 했지?" | → `working/task_m100_86_stage1.md` |
| "어떻게 하기로 했지?" | → `plans/task_m100_86_impl.md` |
| "왜 이 방식으로 하지?" | → `feedback/` + `tech/` |
| "이 함정은 뭐지?" | → `troubleshootings/` |

**작업지시자가 컨텍스트 전달에 소비하는 시간이 거의 0**이다. 이것이 3곳을 이동하며 연속 작업이 가능한 이유이고, 향후 컨트리뷰터가 합류했을 때도 동일하게 작동하는 이유이다.

## 문서 파일명 규칙

모든 문서는 마일스톤 접두어를 포함하여 아카이빙 시 중복을 방지한다.

| 문서 | 파일명 | 예시 |
|------|--------|------|
| 수행 계획서 | `task_{milestone}_{번호}.md` | `task_m100_86.md` |
| 구현 계획서 | `task_{milestone}_{번호}_impl.md` | `task_m100_86_impl.md` |
| 단계별 보고서 | `task_{milestone}_{번호}_stage{N}.md` | `task_m100_86_stage1.md` |
| 최종 보고서 | `task_{milestone}_{번호}_report.md` | `task_m100_86_report.md` |
| 오늘 할일 | `yyyymmdd.md` | `20260409.md` |

## 바이브 코딩과의 차이

| | 바이브 코딩 | Hyper-Waterfall |
|---|---|---|
| AI에게 | "이거 만들어줘" | "이렇게 계획하고, 이렇게 구현하고, 여기서 틀렸으니 이렇게 고쳐" |
| 코드 리뷰 | 안 함 | 매 단계 승인 |
| 문서 | 없음 | 729개 |
| 기술부채 | 축적 | 0에 수렴 |
| 재현 가능성 | 불가능 | 전 과정 타임라인 추적 가능 |
| 결과 | 돌아가긴 하는 코드 | 엔터프라이즈 운영 수준 (9.3/10) |

## 결론

Hyper-Waterfall의 문서 체계는 "문서를 많이 쓰자"가 아니다.

**AI의 기억 한계를 극복하고, 인간의 판단을 보존하고, 전 과정의 추적 가능성을 보장하는 시스템**이다.
